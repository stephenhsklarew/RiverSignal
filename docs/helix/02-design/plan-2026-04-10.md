## 1. Problem Statement and User Impact

Pacific Northwest watershed managers and restoration ecologists spend 15-25 hours/week manually synthesizing data from 5+ disconnected systems (iNaturalist, USGS gauges, state water quality portals, spreadsheets, PDFs) into management decisions. Quarterly funder reports consume 2-3 analyst-days each. Oregon has ~40% fewer working restoration ecologists than a decade ago while citizen-science observation volume grows 20-30% annually. The result: invasive outbreaks detected weeks late, weak outcome evidence threatening grant renewals, and institutional knowledge lost to staff turnover.

RiverSignal closes this gap by providing AI-powered ecological reasoning that transforms fragmented monitoring data into site-level interpretations, restoration forecasts, management recommendations, and funder-ready reports -- all through a map-first workspace.

**If we don't solve it**: Restoration programs continue spending more on data assembly than on ecological decisions. Grant funders lose confidence in outcome evidence. The knowledge gap widens as senior ecologists retire.

**Who benefits**: Watershed program managers (Maria persona -- weekly planning), restoration ecologists (Jake persona -- site interpretation), agency grant managers (Dr. Chen persona -- cross-project evaluation).

## 2. Requirements Analysis

### Functional Requirements

Derived from PRD FR-1 through FR-21 and FEAT-001 through FEAT-006. Organized by subsystem:

**Data Ingestion (FEAT-005)**: Daily automated sync from iNaturalist (by HUC12), USGS Water Services (by station ID), Oregon Water Data Portal (by station); manual upload of CSV/PDF/GeoJSON; canonical schema normalization; site registry with boundaries, intervention logs, data source config; deduplication; health dashboard.

**Ecological Reasoning (FEAT-001, FEAT-002, FEAT-003)**: Site-level ecological summaries with species richness deltas, invasive flags, indicator status, water quality trends, anomaly detection; restoration forecasting with confidence-scored species-return predictions and automated accuracy tracking; ranked management recommendations with seasonal awareness and user feedback loop. All outputs cited to source data.

**Report Generation (FEAT-004)**: OWEB-format quarterly reports auto-generated from site data; PDF and Markdown export; auto-generated maps; editable drafts with data linkage preservation; report versioning.

**Map Workspace (FEAT-006)**: Interactive map with site polygons, color-coded health status, observation overlays; side detail panel with ecological summary, alerts, forecast status, recommendations, and natural-language chat; filtering and layer controls; site boundary definition via drawing or upload.

**Access (PRD FR-19-21)**: Email/password and SSO authentication; organization-level multi-tenancy; role-based access (viewer, analyst, manager, admin).

## 3. Architecture Decisions

### AD-1: Monolith vs. Microservices

- **Question**: Should the MVP be a monolith or distributed microservices?
- **Alternatives**:
  - (A) Microservices: separate services for ingestion, reasoning, API, frontend. Pro: independent scaling. Con: operational complexity for 2-3 person team, network overhead, deployment burden.
  - (B) Modular monolith: single deployable with clear internal module boundaries. Pro: simple deployment, easy debugging, shared database access. Con: must refactor to scale independently later.
  - (C) Two-service split: Python backend (reasoning + ingestion) + Node.js frontend server. Pro: uses each language's strengths. Con: two services to deploy, API contract between them.
- **Chosen**: (B) Modular monolith -- single Python (FastAPI) application
- **Rationale**: 2-3 person team on 90-day timeline cannot afford microservice operational overhead. Python handles all requirements: FastAPI for REST API, background workers for ingestion, Claude SDK for LLM calls, PDF generation. The frontend is a static SPA served from CDN/object storage -- no Node.js server needed. Module boundaries (ingestion, reasoning, api, reports) are enforced by directory structure and import rules, enabling future extraction. **Design for Simplicity** principle: minimal structure that works. **Prefer Reversible Decisions**: monolith-to-services is a well-understood extraction path.

### AD-2: LLM Reasoning Architecture

- **Question**: How should the ecological reasoning engine use the LLM?
- **Alternatives**:
  - (A) Single large prompt: pack all observations + context into one prompt. Pro: simple. Con: hits context limits at ~300 observations; no tool use; fragile.
  - (B) Tool-using agent: LLM with tools for data queries, GIS lookups, and ecological reference retrieval. Pro: scales beyond context window; reasoning is grounded in live data queries; composable. Con: more complex orchestration; latency from multiple tool calls.
  - (C) Map-reduce summarization: pre-summarize data in chunks, then reason over summaries. Pro: handles any observation volume. Con: information loss in summarization; harder to cite specific observations.
  - (D) Hybrid: pre-aggregate data into structured summaries, then pass to LLM with tool access for drill-down queries. Pro: bounded token usage; tool access for specifics; scales well. Con: requires upfront aggregation logic.
- **Chosen**: (D) Hybrid pre-aggregation + tool-using agent
- **Rationale**: Pre-aggregation (species counts, richness deltas, water quality trends) reduces token load to a bounded size regardless of observation count. Tool access allows the LLM to query specific observations, fetch historical comparisons, or look up ecological reference data when needed for reasoning. This keeps prompt costs predictable while maintaining citation quality. The aggregation layer is pure Python (no LLM), which is fast, testable, and deterministic. **Validate Your Work**: aggregation is testable with known inputs/outputs; LLM reasoning is auditable via tool call traces.

### AD-3: Data Storage Strategy

- **Question**: How should ecological data be stored?
- **Alternatives**:
  - (A) Single PostgreSQL with PostGIS: all data in one database with spatial extensions. Pro: simple ops, powerful geospatial queries, mature. Con: time-series performance may require optimization.
  - (B) PostgreSQL + TimescaleDB: add hypertable extension for time-series data. Pro: optimized time-series queries for hydrology/water quality; still single database. Con: additional extension to manage.
  - (C) PostgreSQL + dedicated time-series DB (InfluxDB/TimescaleDB Cloud): separate stores for spatial and temporal. Pro: best-of-breed for each. Con: two data systems to manage; joins across systems are painful.
- **Chosen**: (A) Single PostgreSQL 16 with PostGIS, without TimescaleDB initially
- **Rationale**: MVP data volumes (100K observations, 5 years of daily readings per watershed, ~5 watersheds) are well within PostgreSQL's native capabilities. Standard B-tree indexes on (site_id, timestamp) with BRIN indexes on time-series tables handle the query patterns. TimescaleDB adds operational complexity (extension compatibility, upgrade path) for optimization we don't yet need. If time-series query performance becomes an issue during pilot, TimescaleDB can be added as an extension to the existing PostgreSQL without data migration. **Design for Simplicity**: don't add components for hypothetical scale. **Prefer Reversible Decisions**: adding TimescaleDB later is a non-destructive operation.

### AD-4: Frontend Map Framework

- **Question**: Which mapping library for the map-first workspace?
- **Alternatives**:
  - (A) MapLibre GL JS: open-source fork of Mapbox GL. Pro: free, WebGL-rendered, performant, large community, vector tile support. Con: less polished than Mapbox; self-managed tile hosting.
  - (B) Mapbox GL JS v3: commercial mapping SDK. Pro: polished, great documentation, hosted tiles. Con: usage-based pricing; vendor lock-in; license changed from open source.
  - (C) Leaflet: lightweight raster map library. Pro: simple, well-known. Con: raster-only, poor performance with >1,000 markers, no vector tiles.
  - (D) Deck.gl: high-performance data visualization. Pro: handles millions of points. Con: steeper learning curve, less traditional map UX, overkill for MVP scale.
- **Chosen**: (A) MapLibre GL JS
- **Rationale**: Open-source (BSD-3), WebGL-rendered for smooth performance with 2,000+ markers, vector tile support for custom styling, active community. Free basemap tiles available from multiple providers (MapTiler free tier, Stadia Maps, OpenFreeMap). No vendor lock-in or usage-based costs. PRD specifies MapLibre. **Design for Change**: open-source means we can fork or extend if needed.

### AD-5: Authentication Strategy

- **Question**: How should the MVP handle auth?
- **Alternatives**:
  - (A) Custom auth: build email/password + JWT in FastAPI. Pro: full control, no external dependency. Con: security liability; 2-3 person team shouldn't build auth.
  - (B) Auth-as-a-service (Auth0, Clerk, Supabase Auth): managed auth with SSO support. Pro: secure by default, SSO included, fast integration. Con: external dependency; cost.
  - (C) Supabase Auth + Row-Level Security: auth plus database-level multi-tenancy enforcement. Pro: auth + tenant isolation in one tool. Con: ties auth to database choice.
- **Chosen**: (B) Auth-as-a-service (Clerk or Auth0)
- **Rationale**: Security-critical code should not be hand-built by a 2-3 person team under time pressure. Managed auth provides email/password, SSO, and organization management out of the box. The choice between Clerk and Auth0 is deferred to implementation -- both satisfy requirements. JWT tokens from the auth service are validated by FastAPI middleware. **Design for Simplicity**: don't build what you can buy. **Validate Your Work**: managed auth providers have security certifications the team cannot match.

### AD-6: PDF Report Generation

- **Question**: How should PDF reports with maps and charts be generated?
- **Alternatives**:
  - (A) Server-side HTML-to-PDF (WeasyPrint, wkhtmltopdf): render HTML template then convert. Pro: HTML/CSS is familiar; maps and charts render natively. Con: WeasyPrint CSS support is limited; wkhtmltopdf is unmaintained.
  - (B) Playwright/Puppeteer headless browser: render a full page then PDF print. Pro: pixel-perfect rendering, full CSS/JS support, map tiles render. Con: heavy dependency (headless Chromium); memory and startup overhead.
  - (C) Typst or LaTeX: programmatic document generation. Pro: precise layout control, native PDF. Con: different templating paradigm; map/chart embedding is awkward.
  - (D) React-pdf (server-side): generate PDF directly from React components. Pro: component reuse with frontend. Con: no map rendering; limited styling.
- **Chosen**: (B) Playwright headless browser
- **Rationale**: Reports must include interactive-quality maps (FR-13) and water quality charts (FR-11). Only a browser engine renders MapLibre maps and chart libraries faithfully. Playwright is well-maintained, supports Chromium/Firefox, and has a Python API. The report is first rendered as a web page (reusing frontend components), then PDF-printed via Playwright. Memory overhead is acceptable for batch report generation (not concurrent user requests). Reports are generated asynchronously (background task) with status polling. **Validate Your Work**: rendered output matches what users see in the browser.

### AD-7: Background Task Execution

- **Question**: How should ingestion jobs, report generation, and reasoning tasks run?
- **Alternatives**:
  - (A) Celery + Redis: industry-standard Python task queue. Pro: mature, scalable, retries, scheduling. Con: additional infrastructure (Redis); complexity for MVP.
  - (B) FastAPI BackgroundTasks: built-in background execution. Pro: zero additional infrastructure. Con: no persistence across restarts; no retries; no scheduling; tasks lost on crash.
  - (C) APScheduler + database-backed task queue: Python scheduling library with PostgreSQL as broker. Pro: no Redis dependency; persistent tasks; scheduling built in. Con: less mature than Celery; polling-based.
  - (D) arq (async Redis queue): lightweight async task queue. Pro: simple, async-native, Redis-backed. Con: still requires Redis.
- **Chosen**: (C) APScheduler for scheduling + lightweight PostgreSQL-backed task table for async work
- **Rationale**: MVP needs two things: scheduled jobs (daily ingestion syncs) and async tasks (report generation, reasoning). APScheduler handles cron-like scheduling with PostgreSQL job store (no Redis). For async tasks, a simple `tasks` table with status tracking (pending/running/completed/failed) and a worker loop is sufficient at MVP scale. This avoids adding Redis to the infrastructure. If task volume outgrows this, Celery can be introduced later without changing the task interface. **Design for Simplicity**: no infrastructure beyond PostgreSQL. **Prefer Reversible Decisions**: the task table pattern is compatible with future Celery migration.

### AD-8: API Design

- **Question**: REST or GraphQL for the API?
- **Alternatives**:
  - (A) REST (OpenAPI): standard resource-based API. Pro: simple, well-understood, great tooling, easy caching. Con: multiple round trips for complex views; overfetching.
  - (B) GraphQL: flexible query language. Pro: client specifies exactly what it needs; single endpoint. Con: complexity; caching harder; N+1 query risk; overkill for MVP with one client.
- **Chosen**: (A) REST with OpenAPI specification
- **Rationale**: Single frontend client, well-defined resource boundaries (sites, observations, summaries, reports, recommendations). FastAPI generates OpenAPI spec automatically. No need for GraphQL's query flexibility when the API shapes are known and stable. **Design for Simplicity**.

```json
{
  "error": {
    "code": "SITE_NOT_FOUND",
    "message": "Site with ID 'abc123' does not exist or you lack access.",
    "details": {}
  }

```

HTTP status codes: 400 (validation), 401 (unauthenticated), 403 (forbidden), 404 (not found), 409 (conflict), 422 (unprocessable), 429 (rate limited), 500 (internal), 503 (service unavailable -- e.g., LLM API down).

### Input Validation Rules

- Site boundaries: valid GeoJSON Polygon or MultiPolygon; area < 10,000 km² (prevents accidentally selecting an entire state)
- Date ranges: start < end; max span of 5 years; dates cannot be in the future
- File uploads: CSV max 50 MB; PDF max 100 MB / 50 pages; GeoJSON max 10 MB
- Text fields: max 10,000 characters for intervention descriptions, recommendation notes

- `time_series`: compound index on `(site_id, station_id, parameter, timestamp)`; BRIN index on `timestamp` for range scans

- `intervention`: index on `(site_id, started_at)`
- `recommendation`: index on `(site_id, status, generated_at)`
- `ingestion_job`: index on `(site_id, source_type, started_at)`

### Migration Strategy

Use Alembic for schema migrations. Initial migration creates all tables. Schema is append-only during MVP -- no destructive migrations. Each migration is tested against a seeded database before deployment.

### Multi-Tenancy

Row-level isolation via `org_id` foreign key on `Site`. All queries filter by organization through FastAPI middleware that extracts org from the authenticated user's JWT claims. No shared data between organizations. This is enforced at the query layer, not database level (no RLS for MVP simplicity -- validated by integration tests).

## 6. Error Handling Strategy

### Error Categories

| Category | HTTP Status | Retry | User Message |
|----------|------------|-------|-------------|
| Validation error | 400/422 | No | Specific field-level error messages |
| Auth failure | 401/403 | No | "Please sign in" / "You don't have access to this resource" |
| Resource not found | 404 | No | "{Resource} not found" |
| External API unreachable | 503 | Yes (3x, exponential) | "Data source temporarily unavailable. Using cached data where available." |
| LLM API error | 503 | Yes (2x, 30s backoff) | "Analysis temporarily unavailable. Please try again in a few minutes." |
| LLM rate limit | 429 | Yes (wait for reset) | "System is busy. Your request is queued." |
| LLM timeout (>120s) | 504 | Yes (1x) | "Analysis is taking longer than expected. Retrying..." |
| Database error | 500 | No | "An unexpected error occurred. Our team has been notified." |
| File upload invalid | 422 | No | Specific format/size error with expected format guidance |

### Retry Policies

- **Ingestion syncs**: retry once after 15 minutes; alert admin on second failure; mark source as degraded
- **LLM reasoning calls**: retry twice with 30-second exponential backoff; on third failure, return partial results (data aggregations without LLM narrative) with "AI analysis unavailable" flag
- **Report generation**: retry once; on failure, return error with option to re-trigger
- **File uploads**: no retry (user re-uploads)

### Fallback Behavior

- **LLM unavailable during summary**: return structured data aggregation (species counts, water quality trends) without narrative interpretation. Flag as "automated summary -- AI interpretation unavailable"
- **External API degraded during sync**: continue syncing available sources; flag degraded source on health dashboard; use last successful sync data for reasoning with staleness warning
- **Map tile provider down**: fall back to OpenStreetMap raster tiles (lower quality but functional)

### Structured Logging

All errors logged with: timestamp, request_id, user_id, org_id, error_code, error_message, stack_trace (server-side only), upstream_service (if applicable), retry_attempt.

## 7. Security Considerations

### Authentication & Authorization

- All API endpoints require valid JWT from auth provider (Clerk/Auth0)
- JWT validated on every request by FastAPI dependency injection middleware
- Organization membership and role extracted from JWT custom claims
- Role hierarchy: admin > manager > analyst > viewer
- Permission matrix:

| Action | Viewer | Analyst | Manager | Admin |
|--------|--------|---------|---------|-------|
| View map, summaries, reports | Yes | Yes | Yes | Yes |
| Chat with site | Yes | Yes | Yes | Yes |
| Generate summary/forecast | No | Yes | Yes | Yes |
| Generate report | No | Yes | Yes | Yes |
| Edit report | No | No | Yes | Yes |
| Accept/dismiss recommendations | No | No | Yes | Yes |
| Log interventions | No | Yes | Yes | Yes |
| Upload data | No | Yes | Yes | Yes |
| Create/edit sites | No | No | No | Yes |
| Manage members | No | No | No | Yes |
| Trigger manual sync | No | No | Yes | Yes |

### Attack Surface Analysis

| Surface | Threat | Mitigation |
|---------|--------|------------|
| File upload (CSV/PDF/GeoJSON) | Malicious file content, path traversal, oversized files | Validate file type by content (not just extension); size limits enforced before processing; files processed in isolated temp directories; no user-controlled filenames in storage paths |
| LLM prompt injection via observation data | Adversarial text in iNaturalist observation notes could manipulate reasoning | Observation notes are data-quoted in prompts, not injected as instructions; system prompts use clear instruction/data boundaries; output is always presented as "system interpretation" not as authoritative fact |
| Chat interface | Prompt injection via user queries | User queries are placed in designated user-input slots in the prompt template; system prompt instructs the model to only answer site-related ecological questions; responses are scoped to site data |
| API rate abuse | Excessive LLM calls driving up costs | Per-user rate limits on reasoning endpoints (10 summary requests/hour, 5 forecast requests/hour); per-org daily LLM token budget |
| Data exfiltration across orgs | Accessing another org's data | All queries filtered by org_id from JWT; integration tests verify cross-org isolation; no API endpoint accepts org_id as a parameter (always derived from auth) |
| SQL injection | Malicious input in query parameters | Parameterized queries via SQLAlchemy ORM; no raw SQL string interpolation |

### Data Protection

- All data in transit over TLS 1.3
- Database connections use SSL
- File uploads stored in S3-compatible storage with server-side encryption
- No PII beyond email addresses (managed by auth provider)
- iNaturalist attribution maintained per CC-BY-NC requirements
- API keys (iNaturalist, USGS, Claude) stored in environment variables or secrets manager, never in code or database

### Sensitive Species Data

- The system does not currently handle culturally sensitive species observations (PRD open question)
- If tribal data sovereignty is required, this will need: per-observation access controls, configurable species sensitivity lists, and audit logging of access to sensitive records. This is deferred pending stakeholder input.

## 8. Test Strategy

### Unit Tests

**Coverage targets**: 80%+ on business logic modules; reasoning aggregation layer is the highest priority.

| Module | What to Test |
|--------|-------------|
| Data aggregation (species richness, water quality trends) | Deterministic computation: known observations in → expected deltas out. Edge cases: empty data, single observation, >500 observations, missing prior period |
| Invasive watchlist matching | Match/no-match/low-confidence scenarios against configurable watchlists |
| Indicator species evaluation | Present/absent/new detection logic against site-specific indicator lists |
| Canonical schema normalization | Each source type (iNaturalist, USGS, OWDP, CSV) → canonical record mapping. Malformed input handling. Deduplication logic |
| Recommendation ranking | Priority ordering logic, seasonal calendar evaluation, duplicate filtering |
| Report data assembly | Section assembly from site data, before/after table construction, KPI scoring |
| GeoJSON validation | Valid/invalid polygons, area limit enforcement, overlap detection |
| Auth middleware | JWT validation, role extraction, permission checks |
| Task queue | Status transitions, retry logic, timeout handling |

### Integration Tests

| Scope | What to Test |
|-------|-------------|
| Ingestion → Database | End-to-end sync from mocked external APIs through normalization to database; verify records queryable by site and time range |
| Reasoning pipeline | Pre-aggregation + LLM call (with recorded/mocked LLM responses) → structured output with citations |
| Report generation | Data assembly → template rendering → PDF output (verify PDF contains expected sections) |
| Multi-tenancy isolation | Create 2 orgs, 2 sites; verify org A cannot access org B's data through any API endpoint |
| File upload pipeline | Upload CSV → normalization → queryable observations; upload PDF → text extraction → stored narrative |
| API contract | OpenAPI spec validation; request/response schema conformance |

### End-to-End Tests

| Scenario | Flow |
|----------|------|
| Monday morning check-in | Login → map renders with sites → click site → detail panel shows summary → review recommendations |
| Report generation | Select site → select date range → generate report → poll status → download PDF → verify sections present |
| Invasive alert response | New observation ingested → summary flags invasive → map shows amber alert → manager reviews → assigns action |
| Data upload | Upload CSV → verify observations appear on map → verify included in next summary |

### LLM Testing Strategy

LLM outputs are non-deterministic. Testing strategy:

1. **Golden examples**: Domain advisor validates 10-15 reference ecological summaries. These become regression fixtures. Each release, re-run against recorded inputs and verify outputs are "ecologically reasonable" (manual review, not exact match).
2. **Structural assertions**: LLM outputs must conform to JSON schema (required fields present, confidence scores within range, citations reference existing observation IDs).
3. **Tool call traces**: Log all tool calls during reasoning. Assertions verify the LLM queried relevant data sources (e.g., did it look up water quality for a water quality question?).
4. **Correction rate tracking**: During pilot, domain advisor reviews a sample of outputs weekly. Correction rate is the primary quality metric (<10% for summaries, <20% for forecasts).

## 10. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| iNaturalist observation density in Upper Klamath insufficient for meaningful reasoning | Medium | High | Pre-validate in Phase 2 (W-006): query actual observation counts per HUC12 per quarter. If <50/quarter, supplement with USGS biological datasets or shift flagship to McKenzie watershed which has higher density. Decision gate at end of Week 4. |
| LLM produces ecologically incorrect recommendations that erode user trust | High | High | All reasoning outputs flagged as "AI interpretation -- review before acting." Mandatory domain advisor review sessions weekly during pilot. Golden example regression suite built from advisor-validated outputs. Correction rate tracked as primary quality metric with <10% target. If rate exceeds 20%, pause reasoning features and ship data-only mode. |
| Claude API cost exceeds budget at pilot scale | Medium | Medium | Pre-aggregation reduces token volume by 60-80% vs. raw observation passthrough. Monitor per-request token usage from Week 5. Set per-org daily token budget with admin alerts. If costs exceed $500/month per watershed, optimize prompt templates and increase aggregation granularity. |
| Pilot customer procurement cycle exceeds 90-day timeline | Medium | Medium | Identify 3 pilot candidates in parallel. Offer 60-day free evaluation. Target watershed councils (simpler procurement than state agencies). Have demo-ready environment with synthetic data by Week 8 regardless of customer timeline. |
| PostGIS geospatial query performance degrades with observation volume | Low | Medium | BRIN and GiST indexes designed for access patterns. Performance test at 100K observations per site in Phase 6. If queries exceed 500ms at target volume, add materialized views for map rendering queries. PostgreSQL EXPLAIN ANALYZE on all hot paths before pilot launch. |
| OWEB report format requirements are not well documented | Medium | Medium | Contact OWEB program staff and/or pilot customer in Week 1 to obtain sample reports and format guidelines. If formal spec unavailable, reverse-engineer from 3-5 example reports. Build format as configurable template, not hardcoded, so adjustments are low-cost. |
| Domain advisor availability lapses during critical reasoning development (Weeks 5-8) | Low | High | Front-load knowledge capture: document ecological heuristics, indicator species logic, and seasonal calendars in structured reference files during Weeks 1-2. Build golden example suite from advisor sessions. If advisor is unavailable for >2 weeks, rely on reference files and published ecological literature for PNW riparian systems. |
| iNaturalist API rate limits or terms change | Low | Medium | All ingestion goes through adapter with interface abstraction. Observations cached locally after ingestion -- system never re-fetches processed records. Monitor iNaturalist developer forum for policy changes. If API becomes unavailable, manual CSV upload provides a degraded but functional fallback. |

## 11. Observability

### Logging

- Structured JSON logging via Python `structlog`
- Every log entry includes: `timestamp`, `level`, `request_id`, `user_id`, `org_id`, `service`, `message`
- LLM interactions logged with: `model`, `token_count_input`, `token_count_output`, `tool_calls[]`, `latency_ms`, `cache_hit`
- Ingestion jobs logged with: `source_type`, `site_id`, `records_fetched`, `records_new`, `records_updated`, `duration_ms`, `status`
- Log levels: ERROR for failures requiring attention, WARN for degraded states (stale data, fallback mode), INFO for request lifecycle, DEBUG for development

### Alerting

| Alert | Condition | Severity | Action |
|-------|-----------|----------|--------|
| Ingestion failure | Job fails after retry | Warning | Notify admin via email; mark source degraded on dashboard |
| Ingestion down >24h | Source degraded for >24h | Critical | Notify admin; site summaries show staleness warning |
| LLM API errors | >5 failures in 10 minutes | Warning | Switch to data-only fallback mode; notify admin |
| LLM latency spike | p95 > 180s for 15 minutes | Warning | Investigate; consider reducing aggregation detail |
| API error rate | >5% 5xx responses over 5 minutes | Critical | Page oncall; check database and LLM connectivity |
| Database connection pool exhaustion | Available connections < 5 | Critical | Page oncall; investigate long-running queries |

### Dashboards

- **Operations dashboard**: API request rate/latency, error rate, database connection pool, task queue depth
- **Ingestion dashboard**: per-source sync status, records ingested, failure rate, data freshness
- **LLM dashboard**: request rate by engine, token usage, latency distribution, cost estimate, tool call frequency
- **Product dashboard**: active users, summaries generated, reports exported, recommendation acceptance rate
